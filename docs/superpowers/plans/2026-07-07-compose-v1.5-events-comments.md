# compose v1.5 Events + Comments Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Agents can block on human activity (`compose wait`) and humans can leave anchored comments in the UI that agents read, reply to, and resolve from the CLI.

**Architecture:** A per-doc in-memory event log with long-poll waiters on the existing Fastify server. UI saves and comment activity append events; agent writes do not. Comments persist as JSON under the state dir and broadcast to UI clients over the existing WebSocket.

**Tech Stack:** unchanged (Fastify 5, diff-match-patch, React, Tiptap, Vitest, Playwright).

## Global Constraints

- All v1 global constraints hold (see 2026-07-07-compose-v1.md).
- Event `seq` starts at 1 per doc, session-scoped. Comments persist to `join(stateDir(), 'comments', docId + '.json')`.
- Long-poll `waitMs` default 25000, cap 30000. `compose wait` default timeout 3600 s, exit 0 on events, 2 on timeout.
- Comment ids: 8 hex chars from `randomBytes(4)`.
- Diff summaries: max 12 lines, each snippet capped at 120 chars, prefixed `+ ` / `- `.

### Task A: Event log + diff summary (server pure logic)

**Files:** Create `src/server/events.ts`, `src/server/diffSummary.ts`; tests alongside.

**Interfaces (produces):**
- `summarizeDiff(oldText, newText): string` — diff_main + cleanupSemantic, `+ "..."`/`- "..."` lines, caps per Global Constraints.
- `class EventLog` — `append(docId, event): void` (stamps seq+ts), `since(docId, seq): DocEvent[]`, `latest(docId): number`, `waitFor(docId, seq, waitMs): Promise<DocEvent[]>` (resolves immediately if events exist, else on next append or timeout with `[]`).
- `type DocEvent = { seq; ts } & ({ kind: 'human-edit'; summary } | { kind: 'comment-added'; comment } | { kind: 'comment-replied'; commentId; reply } | { kind: 'comment-resolved'; commentId })`.

**Steps:** failing tests (append/since/latest; waitFor resolves on append; waitFor times out empty; summary shows insertions and deletions, caps length) → implement → PASS → commit `feat: per-doc event log with long-poll waiters and diff summaries`.

### Task B: Comment store (server)

**Files:** Create `src/server/comments.ts`; test alongside.

**Interfaces (produces):**
- Types `Comment`, `CommentReply` per spec.
- `class CommentStore` — `list(docId): Promise<Comment[]>`, `add(docId, {anchorText, body, author}): Promise<Comment>`, `reply(docId, commentId, {body, author}): Promise<CommentReply>` (throws unknown id), `resolve(docId, commentId): Promise<void>`. JSON file per doc under `stateDir()/comments/`, written after each mutation.

**Steps:** failing tests (add persists across a new store instance; reply appends; resolve flips; unknown id throws) → implement → PASS → commit `feat: persistent per-doc comment store`.

### Task C: Server wiring (endpoints, event emission, ws broadcast)

**Files:** Modify `src/server/app.ts`, `src/shared/protocol.ts`; extend `src/server/app.test.ts`.

**Interfaces:**
- Protocol: add `{ type: 'comments'; comments: Comment[] }` to `ServerMsg`.
- Emit `human-edit` after a ws `save` that returns a patch (summary via `summarizeDiff(prevCanonical, newText)` — capture prev text before `handleClientSave`). Watcher external changes emit nothing.
- Endpoints per spec; every comment mutation appends its event AND broadcasts fresh `comments` to the doc's sockets.
- `/restore` also emits `human-edit` (summary of the revert).

**Steps:** failing integration tests (save → events?since=0 has human-edit with summary; external file write → no events; long-poll: GET with waitMs resolves when a save lands mid-wait; comments CRUD round-trip + event kinds + ws `comments` broadcast received) → implement → PASS → commit `feat: events and comments api with live ui broadcast`.

### Task D: CLI wait/comments/reply/resolve

**Files:** Modify `src/cli.ts`; create `src/server/eventsClient.ts` (+ test).

**Interfaces (produces):**
- `pollEvents(port, docId, since, timeoutMs): Promise<{ events: DocEvent[]; latest: number }>` — loops long-polls until events or timeout.
- `formatEvents(events, file): string` — markdown: edits as summary blocks, comments as `> anchor` + body + `id: <id>`; ends with `next: compose wait <file> --since <latest>`.
- CLI subcommands per spec. `compose <file>` behavior unchanged; subcommands ensure server + open doc first (share a helper `openViaApi(file): { port, id }`).

**Steps:** failing tests for pollEvents+formatEvents against a real `buildServer()` (save mid-wait resolves; timeout path) → implement → PASS → manual: two terminals, `wait` unblocks on curl save → commit `feat: compose wait, comments, reply, resolve commands`.

### Task E: UI — selection bubble + comments rail

**Files:** Create `src/client/CommentsRail.tsx`, `src/client/useComments.ts`, `src/client/selectionBubble.tsx`; modify `App.tsx`, `EditorView.tsx`, `useSync.ts` (surface `comments` msgs), `styles.css`.

**Interfaces:**
- `useSync`: new opt `onComments(comments: Comment[])`; controller calls it on `comments` msgs; also fetch initial list via GET on mount (App does it).
- `CommentsRail` props: `{ docId, comments, editor, pendingAnchor, onClearPendingAnchor }` — compose box appears when `pendingAnchor` set; posts author 'you'; reply input per thread; resolve button; unanchored badge when `editor` text lacks anchorText; click → scroll+flash (search doc text, `setTextSelection`, temp decoration via existing highlight meta).
- Selection bubble: on `selectionUpdate` with nonempty selection, absolutely-positioned button at `view.coordsAtPos(from)`; click → `onComment(selectedText)` → sets `pendingAnchor` in App.
- App layout becomes 3 columns (sidebar | doc | rail 280px, rail hidden when no comments and no pendingAnchor).

**Steps:** implement → typecheck+build → manual smoke via dev server → commit `feat: selection comments with live threaded rail`.
